Where it came from

My wife runs a small ballet school in rural Australia. Managing class schedules, student enrolments, and invoicing is a genuine pain — existing tools are either too generic or priced for large studios. Class Koala started as a solution to a real problem, and I'm treating it as if I intend to sell it one day. That constraint forces decisions that a toy project never would: how do you safely separate data between schools? How do you handle billing per tenant? What does the domain model actually look like when the problem is real?

Whether it goes anywhere commercially is an open question — but building with that ambition has been the point.

What I set out to explore

Multitenancy

Row-level tenant isolation using scoped queries and middleware, ensuring data separation between schools/organisations without separate databases.

Real-time Broadcasting

ActionCable with Turbo Streams to push live updates to connected clients — no page refreshes, no polling.

Data Modelling

Modelling a domain from scratch is its own discipline. I've done data modelling before, but greenfield design — deciding what entities exist, how they relate, and what constraints matter — surfaces different questions than working within an existing schema.

Tech stack

Rails 8 Ruby 3.2 PostgreSQL Hotwire — Turbo + Stimulus ActionCable TailwindCSS ViewComponent

Note: This project is intentionally incomplete — it's a learning vehicle built in my own time, not a finished product. The goal was to build something with real stakes: a genuine domain, real users (eventually), and production deployment. That context forces better decisions than a tutorial ever could. It's ongoing.
