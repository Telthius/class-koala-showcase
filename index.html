Where it came from

My wife runs a small ballet school out in rural Australia. Keeping track of class schedules, who's enrolled in what, and chasing invoices is a real headache. The tools that exist are either too generic or built for large studios with budgets to match. So I started building something.

I decided early on to build it like I actually intended to sell it one day. That changes the questions you ask. How do you keep one school's data away from another's? How does billing work per tenant? What should the data model actually look like when the problem is real and not made up for a tutorial? Whether it ever becomes a product is genuinely uncertain, but that's kind of the point.

What I set out to explore

Multitenancy

Each school gets its own isolated slice of the database using row-level scoping. One app, one database, but each tenant only ever sees their own data.

Real-time Broadcasting

Using ActionCable with Turbo Streams to push updates to the page without a refresh. It's something I'd read about plenty but hadn't actually built before.

Data Modelling

I've worked with existing schemas plenty but designing one from scratch is a different thing. Figuring out what the entities actually are, how they relate and where the edges of the domain sit has been one of the more interesting parts of this project.

Tech stack

Rails 8 Ruby 3.2 PostgreSQL Hotwire (Turbo + Stimulus) ActionCable TailwindCSS ViewComponent
